# Designing the 404 page

Never ship a generic "404 Not Found" page — it must inherit the site's actual
visual language (same fonts, palette, layout chrome, tone of voice) so a wrong URL
doesn't break the experience.

## Required elements
- Same header/nav and footer as the rest of the site (or a deliberately minimal
  version of them) — never a blank white page with just an error string.
- A clear, plain-language statement that the page wasn't found — no raw error
  codes as the primary message (NN/g heuristic #9: help users recognize/recover
  from errors).
- A way back: a link to the homepage and, where relevant, to the most likely
  intended destination (e.g. search, or top-level nav).
- On-brand tone: if the site's voice is playful, the 404 copy can be too — but
  never at the cost of the "how do I get back" action being clear.
- Correct HTTP status: serve an actual `404` status code, not a `200` with a
  "not found" page (a common SEO/crawler-hygiene mistake).

## Implementation notes
- Static hosts: most (Netlify, Vercel, GitHub Pages) auto-serve a `404.html` at the
  site root with a proper status code — name the file `404.html` and place it at
  the root of the deployed output.
- Reuse the site's actual CSS/layout partials rather than duplicating styles into
  a standalone file that will drift out of sync over time.
- Keep it lightweight — this is the worst possible page to be slow or heavy on,
  since the visitor already hit a dead end.
